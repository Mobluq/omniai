# Architecture Decision Records

Use ADRs for decisions that shape future development. A good ADR is short, dated, and clear enough that a future engineer can understand why the team chose one path over another.

## When to Add an ADR

Add an ADR when a change affects:

- application architecture
- database ownership or migration strategy
- security posture
- deployment or CI/CD process
- major dependencies
- public API contracts
- long-term product constraints

## Format

Use a numbered file:

```text
0005-short-decision-title.md
```

Each ADR should include:

- Status
- Context
- Decision
- Consequences
